# Telegram: использование реального пользователя `Adam`

## Контекст

В этой среде у агента есть доступ не только к Bot API, но и к реальному Telegram-пользователю.

- Пользовательский аккаунт: **Adam**
- Это основной «человеческий» Telegram-аккаунт агента в `q tg`/Telethon-связке.

## Когда использовать реального пользователя (а не бота)

Использовать `Adam`, если задача требует возможностей, которые часто ограничены у ботов:

- операции, завязанные на пользовательские права/контекст диалога;
- действия в чатах/каналах, где бот не может выполнить нужный edit/entity;
- кейсы с Telegram entity/редактированием, где поведение Bot API и user API различается;
- доступ к данным/диалогам, которые практически удобнее через user-клиент.

## Базовый принцип выбора

1. **Сначала** обычный безопасный путь (бот/скрипт).
2. Если упёрлись в ограничение бота/бот-токена — **переключаемся на `q tg` под пользователем `Adam`**.
3. После действия — верифицируем результат чтением сообщения/чата.

## Мини-плейбук

1. Прочитать нужный chat/message через `q tg raw`.
2. Выполнить действие через Telethon (`tl_call`/`raw`, в зависимости от сценария).
3. Повторно прочитать объект и проверить, что изменения реально применились.
4. Сообщить пользователю, что сделано, и дать ссылку/ID.

## Важные ограничения

- Не выполнять внешние/публичные действия без явного запроса пользователя.
- Не делать массовые/разрушительные действия без подтверждения.
- Всегда учитывать риск гонок (параллельные запуски cron/manual).

## Короткая памятка

- `Adam` в Telegram = доступный «реальный пользовательский» путь для задач, где бот ограничен.
- Этот путь допустим и может использоваться по необходимости как обычный рабочий инструмент.
